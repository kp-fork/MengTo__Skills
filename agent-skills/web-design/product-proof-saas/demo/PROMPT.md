# Product Proof SaaS Demo Prompts

## Minimal prompt

Use $product-proof-saas to create an original SaaS landing page where an honest, deterministic workflow demo is the central proof and every generated or sample state is clearly labeled.

## Recreate the demo

Use $product-proof-saas to recreate the design quality and behavior of the included Draftline editorial-workflow demo as a complete standalone `index.html`.

Build a white-to-pale-blue atmospheric shell with soft grid lines, near-black type, compact mono labels, one cool signal color, thin borders, and 12–16px radii. Place a concise friction-led headline on the left and a dense product stage on the right. The product stage should show editable input, a deterministic sample output, a selected direction, version context, progress, and an explicit `Sample · not published` label.

Make the run action visibly transition through processing and ready states without pretending to contact a model. Keep input, processing, draft, edit, approval, and publish concepts understandable without animation. Support keyboard editing, submit, cancel or retry where relevant, loading, quota, moderation, network, and error states. Disable autoplay typing for reduced motion.

Carry the workflow into focused feature modules, audience views, an honest pricing comparison, and deep-linked FAQ terms. Use only local HTML, CSS, JavaScript, and fictional sample data. Do not invent customers, generation speed, outcomes, security claims, or usage limits.

## Remix prompt

Use $product-proof-saas and the included demo as the quality bar for a different product category. Replace the brand, workflow, copy, palette details, sample data, and feature set. Preserve the product-as-proof hierarchy, explicit state model, honest sample labeling, restrained motion, accessible controls, responsive composition, and reduced-motion fallback.
